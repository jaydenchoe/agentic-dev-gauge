Core

명사 · noun

도메인 모델 + Port 정의. 외부 라이브러리 import 없음. 같은 입력 → 항상 같은 출력.

Service

동사 · verb

흐름 조율. Port를 통해 외부와 협력. 세상의 중심. 세상이 Service에 맞춰야 한다.

Port

씬 레이어 · thin

ABC 인터페이스만. 구현 코드 없음. "이 모양으로 줘야 해"라는 계약서.

Adapter

외부 접착제 · glue

Port를 구현 + 외부 라이브러리 사용. Port 하나에 어댑터 여러 개 붙을 수 있음.








  
핵심 인사이트

  

    

      🎯
      

        의존성은 항상 안쪽으로
        Core는 아무것도 모른다. Adapter가 Port를 알고, Port가 Core를 안다. 방향이 뒤집히면 Hexagonal이 깨진다.
      

    

    

      📜
      

        Port는 계약서, 코드가 아니다
        Port가 두꺼워지면 경계가 흐려진다. Port는 얇을수록 Hexagonal이 건강하다는 신호다.
      

    

    

      🔌
      

        Port 하나 → Adapter 여러 개
        LocalLlmPort 하나에 OllamaAdapter와 LMStudioAdapter가 붙는다. Service는 어떤 Adapter인지 모른다.
      

    

    

      🧭
      

        아키텍처는 발견이다
        "Make it work → Make it right." 처음엔 뭉뚱그려 만들다가 코드가 말해줄 때 경계를 발견한다.
      

    

    

      🧪
      

        Core가 순수할수록 테스트가 쉽다
        Core 테스트에 mock이 필요하면 Core에 있으면 안 되는 게 섞여 있다는 신호다.
      

    

    

      🌍
      

        세상이 Service에 맞춰야 한다
        psutil이 macmon으로 바뀌어도, claude.ai HTML이 바뀌어도 Service/Core는 흔들리지 않는다.
      

    

  







  
Hexagonal vs Clean Architecture

  

    

      
Hexagonal (Ports & Adapters)

      
    
        
  • 창시자: Alistair Cockburn (2005)
    • 
        
  • 구조: 내부 / 외부 2분법
    • 
        
  • 핵심: 경계 (Port 안쪽 vs 바깥쪽)
    • 
        
  • 안쪽 구조화 방식은 자유
    • 
        
  • 진입장벽 낮음
    • 
      

    

    

      
Clean Architecture (Uncle Bob)

      
    
        
  • 창시자: Robert C. Martin (2012)
    • 
        
  • 구조: Entity → UseCase → Interface → Framework 4레이어
    • 
        
  • 핵심: 경계 + 레이어 순서 엄격히
    • 
        
  • Hexagonal에서 영향 받음
    • 
        
  • 진입장벽 높음, 소규모엔 과함
    •
